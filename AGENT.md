## AGENTS 指令

- 流式输出：所有回复尽可能以逐步（流）形式呈现，比如分段、分点或者分段输出，让用户感觉到正在连续生成内容。若用户反馈阅读受影响，可切换为"阶段整块发送"，按阶段或关键节点一次性输出完整段落。
- 语言：全部使用中文，避免混入其它语言，思考的时候也用中文。
- 其余指令：遵循默认运行环境，不要假设自动发送命令。
- 第一性原理：多使用第一性原理，没用的话少说。
- 流程表达：说之前想一遍自己的中文是否是优秀的清晰的表达，同时分段落有空行的数据，不要字挤在一起。
- 任务表：多步任务或用户期望时，优先用 `update_plan` 建立并维护 pending/in_progress/completed 的步骤清单，执行过程中及时更新状态并流式汇报；显然单步即可完成的简单请求可直接完成但需保持流式说明。
- 自查测试：涉及代码修改的任务，默认在任务表或执行流程中加入"自查测试/快速验证"步骤（如最小化自测、lint/单测，按上下文选择），除非用户明确表示不需要。

## 执行模式与安全红线

### 执行模式（会话级）
- **默认审慎模式**：需逐步确认方案与风险
- **开启coding模式**：用户明确说开始信号后，在已确认方案与范围内，允许直接改代码/写temp脚本/跑必要测试且不重复YES/NO
- **关闭coding模式**：用户说"关闭coding模式"或会话结束后恢复审慎模式

### 严禁要求（必须单独确认）
1. **Git操作**：回溯版本前必须询问×3
2. **Python环境**：得到许可后再行动
3. **Coding模式例外边界**（开启coding模式后仍需单独确认）：
   - Git回溯/reset/强推/删分支等破坏性操作
   - 安装新库、变更任何依赖版本、切换Python环境
   - 涉及密钥/账户/权限/生产部署的操作
   - 超出已确认范围的大改动
4. **核心依赖锁定**：变更任何版本前必须询问（见下方项目上下文中的依赖列表）

## 编码与中文安全规则（必须遵守）

- 修改任何含中文文件前，先确认原文件编码（UTF-8/GBK/ANSI），不得猜测。
- 必须保持原编码和原换行风格写回，禁止无确认的整文件重写。
- 优先使用最小化补丁（apply_patch），避免使用整段覆盖写入导致编码漂移。
- 若发现中文显示异常（乱码），立即停止修改并先报告原因，再继续处理。
- 若需统一编码，必须作为独立任务处理，不与业务逻辑修改混在同一次变更里。
- **读取大文件前必须检查大小**：读取 CSV/Parquet/任何数据文件前，先用 ls -lh 或 stat 检查文件大小，大于 1GB 的文件禁止直接用 pandas 读取，必须用流式读取或只读取元数据/前几行。
- **临时文件放到 temp/ 文件夹里**。

### 报价规则约定
- 不干胶报价页已引入“全局价格缩放系数”参数，对应前端输入项 `priceScale`，默认值为 `1`。
- 所有报价规则先按原始规则计算出基础价格，再在最终输出前统一乘以该系数。
- 示例：缩放系数为 `0.8` 时，所有报价结果都应为原价的 `0.8` 倍。
- 后续新增任何产品或工艺报价规则时，都应复用这一统一缩放出口，不要把缩放逻辑分散写进各条局部规则中。
- 当前“铜板纸不干胶”规则已改为“通用公式优先”的实现，不再按 `5*5`、`5*10` 这类固定尺寸整体写死。
- 现阶段的通用公式实现方式：
  - `彩色印刷+烫金/银` 已切到“二维尺寸公式”：基于 `(宽, 高)` 样本点做二维插值，而不是只看面积档。
  - 该二维公式会先把尺寸排序成 `(短边, 长边)`，避免 `5*10` 与 `10*5` 算出不同结果。
  - 当输入尺寸恰好等于已知样本点时，结果应精确命中该样本值；其它尺寸走同一套插值公式。
  - 面积档位以 `25cm²` 为一档的逻辑，当前只继续用于非 `彩色印刷+烫金/银` 的旧推导分支。
  - `1*1` 这类超小尺寸单独走 `micro` 档。
  - `覆亮膜 / 无附加工艺`：按面积档直接取档位价。
  - `覆亮膜 / 彩色印刷+烫金银`：按二维尺寸公式计算。
  - `覆哑膜 / 彩色印刷+烫金银`：按二维尺寸公式计算。
  - `纯烫金`、`烫镭射银`：基于“无附加工艺”和“彩色印刷+烫金银”的价差比例推导。
  - `覆哑膜`：基于对应亮膜价格乘既有样本推导出的倍率。
- 这套规则的统一入口在 [static/app.js](/D:/lu_store/static/app.js)。
